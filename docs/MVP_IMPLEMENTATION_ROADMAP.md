# Sber A2A Procurement Platform — пошаговый план демонстрационного MVP

> Superseded roadmap: сохранён как история. Текущая топология определена ADR-0007.

## 1. Назначение документа

Этот документ описывает последовательный путь от исходного состояния проекта до
демонстрационного MVP платформы агентных B2B-закупок. Фактически реализованное
состояние MVP зафиксировано отдельно в `docs/IMPLEMENTED_MVP.md`.

Итоговый MVP не обязан подключаться к реальным системам Сбера, ERP, ЭДО, KYB/AML
или платёжному контуру. При этом архитектура должна быть готова к таким
подключениям: временная реализация каждой внешней системы оформляется отдельным
mock-адаптером с тем же контрактом, который позже сможет реализовать production-
интеграция.

Документ является планом реализации. Этапы выполняются последовательно. Переход
к следующему этапу допускается после выполнения критериев готовности предыдущего.

---

## 2. Что считается готовым демонстрационным MVP

MVP считается готовым, если на одной машине через Docker Compose можно запустить:

- frontend;
- независимый агент клиента A1;
- оркестратор A3;
- три независимых агента поставщиков A2.1, A2.2 и A2.3;
- PostgreSQL;
- при необходимости локальный брокер фоновых задач.

Пользователь должен иметь возможность:

1. Создать закупочную потребность из формы или свободного текста.
2. Увидеть новую сделку в истории.
3. Наблюдать в реальном времени:
   - получение A3 запроса от A1;
   - проверку мандата;
   - обнаружение поставщиков;
   - отправку отдельных RFQ каждому A2;
   - состояния задач поставщиков;
   - получение оферт или ошибок;
   - проверку ограничений;
   - расчёт рейтинга.
4. Открыть каждое переданное сообщение и увидеть его метаданные.
5. Сравнить допустимые и отклонённые оферты.
6. Подтвердить выбранную оферту от имени авторизованного пользователя.
7. Получить один заказ и один черновик платежа.
8. Повторить запрос подтверждения и убедиться, что второй заказ не создан.
9. Перезапустить A3, обновить браузер и продолжить работу с сохранённой сделкой.
10. Увидеть отправку award выбранному поставщику и уведомлений остальным.
11. Увидеть mock-статусы исполнения заказа вплоть до завершения демонстрационной
    поставки.

Ключевой критерий: взаимодействие A1, A3 и A2 должно происходить между
независимыми сервисами по сетевому контракту, а не через вызовы Python-методов
внутри одного процесса.

## 3. Исходное состояние — точка А

Этот раздел описывает baseline до реализации сетевого и persistent MVP. Он
оставлен как историческая точка сравнения с текущим состоянием из
`docs/IMPLEMENTED_MVP.md`.

В проекте уже реализованы:

- FastAPI backend;
- React/TypeScript frontend;
- доменные Pydantic-модели;
- детерминированная проверка части ограничений;
- детерминированный ranking;
- LangGraph workflow;
- REST API и MCP-интерфейс A3;
- три локальных mock-поставщика;
- подтверждение человеком;
- генерация `order_id` и `payment_draft_id`;
- временный список событий сделки;
- базовые backend- и frontend-проверки.

Текущие ограничения:

- A1 не является отдельным агентом;
- A2 создаются как локальные Python-объекты внутри A3;
- реального сетевого A2A-обмена нет;
- все сделки хранятся в памяти;
- история списка сделок отсутствует;
- события не являются отдельным append-only ledger;
- workflow выполняется внутри одного долгого HTTP-запроса;
- frontend получает результат только после завершения workflow;
- подтверждающий пользователь определяется строкой из запроса;
- заказ и платёжный черновик представлены только UUID;
- нет timeout, retry, inbox/outbox и устойчивой идемпотентности;
- внешний вызов LLM может задержать основной flow;
- Agent Card A3 пока является информационным JSON, а не частью полного
  протокольного взаимодействия.

## 4. Принципы реализации MVP

### 4.1. Настоящее ядро, временные интеграции

В MVP должны быть настоящими:

- доменная модель сделки;
- state machine;
- сетевое взаимодействие агентов;
- хранение состояния;
- история сделок;
- Deal Ledger;
- hard constraints;
- ranking;
- проверка полномочий внутри demo-контура;
- human-in-the-loop;
- идемпотентность;
- обработка ошибок и таймаутов;
- восстановление после перезапуска;
- API-контракты;
- наблюдаемость.

Временными mock-адаптерами могут быть:

- ERP;
- корпоративный каталог;
- IAM/SSO;
- Mandate Service;
- KYB/AML;
- supplier risk service;
- ЭДО;
- платёжный контур;
- доставка;
- уведомления.

### 4.2. Mock не должен проникать в доменное ядро

Неправильно:

```python
if demo_mode:
    order_id = uuid4()
else:
    call_sber_api()
```

Правильно:

```python
class OrderGateway(Protocol):
    async def create_order(self, command: CreateOrderCommand) -> Order: ...


class MockOrderGateway(OrderGateway):
    ...


class SberOrderGateway(OrderGateway):
    ...
```

Доменный сервис вызывает `OrderGateway` и не знает, какой адаптер подключён.

### 4.3. LLM не принимает решений

LLM разрешено использовать только для:

- преобразования свободного текста в черновик;
- формирования понятного объяснения готового рейтинга.

Ошибка, timeout или отсутствие LLM не должны останавливать закупочный workflow.

### 4.4. Один источник истины

PostgreSQL является источником истины для состояния сделки. LangGraph checkpoint
используется для восстановления выполнения, но не заменяет доменную БД и Deal
Ledger.

### 4.5. Все внешние вызовы ненадёжны

Каждый сетевой вызов должен иметь:

- timeout;
- ограниченный retry;
- correlation ID;
- idempotency key;
- журналирование результата;
- контролируемую обработку частичного отказа.

## 5. Целевая архитектура MVP

```text
┌─────────────┐       A2A/HTTP       ┌──────────────────────────┐
│ A1 service  │ ───────────────────► │ A3 gateway/orchestrator  │
└──────┬──────┘                      └────────────┬─────────────┘
       │                                         │
       │                                         ├──► A2.1 service
       │                                         ├──► A2.2 service
       │                                         └──► A2.3 service
       │
┌──────▼──────┐                         ┌────────────────────────┐
│ React UI    │ ◄──── REST + SSE ─────► │ PostgreSQL             │
└─────────────┘                         │ deals, tasks, messages,│
                                      │ quotes, events, orders  │
                                      └────────────────────────┘

Mock adapters inside defined integration boundary:

A1  ─► Mock ERP / Mock Mandate Provider
A2  ─► Mock Supplier Catalog
A3  ─► Mock IAM / Risk / KYB / EDO / Payment / Notification
```

Рекомендуемое разделение каталогов:

```text
src/
  procurement_core/       # Доменные модели и правила без FastAPI и mock-кода
  a3_service/             # Оркестратор и входной A2A/REST API
  a1_service/             # Демонстрационный агент клиента
  a2_service/             # Переиспользуемый runtime агента поставщика
  integrations/
    contracts/            # Порты/Protocol-интерфейсы
    mock/                 # Временные реализации
    production_stubs/     # Заготовки конфигурации реальных адаптеров
  infrastructure/
    persistence/
    messaging/
    observability/
```

На первом этапе допустимо сохранить один Python package, но процессы A1, A3 и
A2 должны запускаться отдельными entrypoint-командами.

  - `ApprovalDecision`;
  - `Award`;
  - `Order`;
  - `PaymentDraft`;
  - `FulfillmentStatus`.
- [ ] Определить envelope каждого сообщения.
- [ ] Добавить JSON Schema для структурированных payload.
- [ ] Определить state machine сделки и допустимые переходы.

### Минимальный message envelope

```json
{
  "message_id": "uuid",
  "message_type": "sber.procurement.rfq.v1",
  "schema_version": "1.0",
  "correlation_id": "deal uuid",
  "causation_id": "previous message uuid",
  "sender_agent_id": "a3-sber-demo",
  "recipient_agent_id": "supplier-a2-1",
  "created_at": "ISO-8601",
  "idempotency_key": "stable key",
  "payload": {}
}
```

### State machine MVP

```text
draft
  → validating_mandate
  → discovering_suppliers
  → collecting_quotes
  → evaluating_quotes
  → awaiting_approval
  → creating_order
  → order_created
  → fulfillment_in_progress
  → completed

Terminal/alternative:
  rejected
  canceled
  failed
  expired
```

### Промежуточные задачи

- [ ] Добавить отдельные статусы A2A task.
- [ ] Разделить внутренний `deal_id`, внешний `task_id` и `message_id`.
- [ ] Убрать риск поставщика из данных, контролируемых самим поставщиком.
- [ ] Зафиксировать, включён ли НДС в цену.
- [ ] Добавить единицу измерения.
- [ ] Зафиксировать формат денежных значений без `float`.

### Критерий готовности

- Все схемы имеют версию.
- Для каждого сообщения существуют сериализация, валидация и contract test.
- Доменный пакет не импортирует FastAPI, LangGraph или mock-поставщиков.

## 9. Контрольные демонстрационные сценарии

## Сценарий 1. Основной flow

1. Buyer входит в demo-контур.
2. A1 создаёт потребность на 20 подшипников.
3. A3 проверяет мандат.
4. A3 обнаруживает три A2.
5. Все A2 получают RFQ.
6. Все A2 возвращают оферты.
7. A3 детерминированно сравнивает предложения.
8. Approver подтверждает оферту.
9. Создаются Order и PaymentDraft.
10. Выбранный A2 получает award.
11. Остальные получают rejection.
12. Mock fulfillment доводит сделку до `completed`.

## Сценарий 2. Частичный отказ

1. A2.2 настроен на timeout.
2. A2.1 и A2.3 возвращают валидные оферты.
3. A3 ждёт до RFQ deadline.
4. Сделка продолжает работу с двумя офертами.
5. В ledger видны timeout и retry A2.2.

## Сценарий 3. Отклонённая оферта

1. Один A2 возвращает неподходящий SKU или превышает бюджет.
2. Quote сохраняется как evidence.
3. Quote получает `eligible=false`.
4. UI показывает точную причину отклонения.

## Сценарий 4. Идемпотентность

1. Approver дважды отправляет подтверждение.
2. Сеть повторяет запрос.
3. A3 возвращает исходный Order.
4. В БД существует один Order и один PaymentDraft.

## Сценарий 5. Восстановление

1. Сделка находится в `collecting_quotes`.
2. A3 останавливается.
3. A3 запускается снова.
4. Workflow восстанавливается.
5. Полученные ранее сообщения не дублируются.

## Сценарий 6. Недоступный OpenRouter

1. LLM включён, но provider не отвечает.
2. Срабатывает timeout.
3. Используется детерминированное объяснение.
4. Основной flow завершается успешно.
